# speculative_token_tree 实现方案与验证报告

## 一、参数说明

`speculative_token_tree` 是 `--speculative-config` 的子参数，用于配置 EAGLE 推测解码的 tree 结构。

### 与原生 vLLM 的关系

原生 vLLM 已完整支持 `speculative_token_tree`：
- `vllm/v1/attention/backends/tree_attn.py` — GPU tree attention backend
- `vllm/v1/spec_decode/eagle.py` — `propose_tree` 方法

NPU 需要实现对应的 backend 和适配。

---

## 二、NPU 实现方案

### 2.1 核心差异

| 层面 | GPU (原生 vLLM) | NPU (本实现) |
|------|----------------|-------------|
| Attention kernel | FlashAttention | `npu_fused_infer_attention_score` |
| Mask 格式 | float32 (0=attend, -inf=block) | int8 (0=attend, 1=block) |
| Mask 尺寸 | 原始 tree_len × tree_len | pad 到 2048×2048 |
| Mask 缓存 | 无 | 预计算所有 slice mask |
| ACL Graph | CUDA Graph | `ACLGraphWrapper` + `cudagraph_dispatcher` |
| actual_seq_lengths | tensor | tensor (优化后，之前为 `.tolist()`) |

### 2.2 文件结构

```
vllm_ascend/
├── attention/
│   ├── backends/
│   │   ├── __init__.py          # backends 包初始化
│   │   └── tree_attn.py         # AscendTreeAttentionBackend
│   ├── attention_v1.py          # 注册 TREE_ATTN backend
│   └── attention_mask.py        # mask 转换工具
├── spec_decode/
│   └── eagle_proposer.py        # propose_tree + _propose_tree
└── ascend_forward_context.py    # slot_mapping 参数
```

### 2.3 AscendTreeAttentionBackend

```python
class AscendTreeAttentionBackend(AttentionBackend):
    get_name()         → "TREE_ATTN"
    get_impl_cls()     → AscendTreeAttentionImpl
    get_builder_cls()  → AscendTreeAttentionMetadataBuilder
```

### 2.4 AscendTreeAttentionMetadataBuilder

- 解析 `speculative_token_tree` 配置
- 构建 GPU float32 tree mask → 转换为 NPU int8 mask
- 预计算所有 slice mask（`_precompute_slice_masks`）
- `build_for_drafting`: 使用缓存的 mask，避免运行时重复计算

### 2.5 propose_tree / _propose_tree

- `_propose_tree`: 第一次 draft model forward + compute logits + 调用 `propose_tree`
- `propose_tree`: tree 层级循环，每层调用 draft model forward
- 每层支持 ACL Graph dispatch（`cudagraph_dispatcher.dispatch`）

### 2.6 _forward_decode 优化

优化后的 `_forward_decode` 方法支持两种模式：

1. **ACL Graph 模式**（`_EXTRA_CTX.is_draft_model = True`）:
   - 预计算 workspace（`_npu_fused_infer_attention_score_get_max_workspace`）
   - 使用 `.out()` 变体避免额外内存分配
   - 与标准 decode 的 `full_graph_fia` 模式对齐

2. **普通模式**（`_EXTRA_CTX.is_draft_model = False`）:
   - 直接调用 `npu_fused_infer_attention_score`
   - 保持原有逻辑不变

两种模式都使用 tensor 类型的 `actual_seq_lengths`（`.to(torch.int64)`），避免 CPU-GPU 同步。

## 六、进一步优化尝试（优化 4-6，已回退）

### 6.1 优化 4：ExternalEvent 同步机制

**尝试内容**: 在 `_forward_decode` 的 ACL Graph 分支中添加完整的 ExternalEvent 同步机制（`event.wait/reset` + `graph_task_group_begin/end`），与 `full_graph_fia` 对齐。

**E2E 测试结果**:
| 模式 | 优化前 | 优化 4 后 | 变化 |
|------|--------|----------|------|
| Tree n=2 | +14.3% | +12.0% | **-2.3pp** |
| Tree n=3 | +10.2% | +5.3% | **-4.9pp** |

**结论**: ExternalEvent 同步机制在 tree attention 场景下引入了额外的同步开销，反而降低了性能。已回退。

### 6.2 优化 5：启用 Async Scheduling

**尝试内容**: 移除 `platform.py` 中强制禁用 async scheduling 的代码（因为 `update_num_computed_tokens_for_batch_change` 已实现）。

**E2E 测试结果**: 与优化 4 一起测试，同样有负面影响。

**结论**: Async scheduling 与 tree speculative decoding 的交互产生了额外的调度开销。已回退。

### 6.3 优化 6：减少 mask padding

**调研结果**: NPU kernel 对 sparse_mode=3 要求 mask 必须是 `(2048, 2048)`，这是硬件/驱动层的硬性约束，无法改变。

**结论**: 不可行。

---

## 七、最终结论

当前 NPU tree speculative decoding 性能：
- **Tree n=2**: +14.3% vs Baseline（CUDA: +15.6%，差距 1.3pp）
- **Tree n=3**: +10.2% vs Baseline（CUDA: +13.2%，差距 3.0pp）

有效的优化（已完成）：
1. 消除 `.tolist()` CPU 同步 → +1.9pp
2. ACL Graph workspace 预计算 + `.out()` 变体
3. Mask 缓存优化

不可行的优化：
- ExternalEvent 同步：引入额外开销
- Async Scheduling：与 tree 模式交互有负面影响
- Mask padding：硬件约束无法改变

**NPU 与 CUDA 的差距已缩小到 1.3pp（tree n=2），进一步优化空间有限。**
