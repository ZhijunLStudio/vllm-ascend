| 功能完整性 | 30% | 是否完全实现预期目标 |

| 创新性 | 5% | 算法或工程上的独特优化 |

## 候选参数分析

### 已排除的参数

| 参数 | 排除原因 |
|------|----------|
| torchao | 创新性低 - 依赖 torchao 库，vLLM 层只是薄适配层（~50行），创新不在 vLLM-Ascend |
| gguf | **致命风险** - 依赖自定义 CUDA kernel（ggml_dequantize, ggml_mul_mat_a8 等），NPU 无等价算子。W5 格式 torch_npu 不支持。需要写大量 C++ 算子 |
| mamba-block-size | 代码量太小 - 仅需修改 2 个 patch 文件（~100行），不符合"高"难度预期 |
| use_prefill_query_quantization | 代码量太小（~300行）+ 依赖 kernel 支持不可控 + 2队竞争 |
| dflash | 无上游实现参考，风险极高 |
| lmcache | 4队竞争，太卷 |
| awq | 4队竞争，太卷 |

### 最终候选：两个参数

#### 候选1：--cpu-offload-params（编号22）

**现状：**
- vLLM 上游有完整的 UVA/Prefetch offloader 框架（`vllm/model_executor/offloader/`）
- Ascend 当前仅有独立的 `weight_prefetch.py`（45行）和 `patch_uva.py`（85行）
- `model_runner_v1.py` 中 `offload_config.uva.cpu_offload_gb` 被 assert=0，offloader 未被调用
- 完全没有接入 vLLM 的 OffloadConfig 参数体系

**需要实现：**
1. 新建 `vllm_ascend/model_executor/offloader/__init__.py` — 注册 NPU offloader（~50行）
2. 新建 `vllm_ascend/model_executor/offloader/uva.py` — NPU UVA 管理器，替换 CUDA 流/内存为 torch.npu（~200行）
3. 新建 `vllm_ascend/model_executor/offloader/prefetch.py` — NPU 预取管理器，集成 npu_prefetch（~300行）
4. 新建 `vllm_ascend/worker/buffer_utils.py` — NPU UvaBuffer，替换 CUDA caching allocator（~150行）
5. 修改 `model_runner_v1.py` — 移除 assert=0，接入 offloader 框架（~200行）
6. 修改 `patch_uva.py` — 和新框架整合（~100行）
7. 修改 `weight_prefetch.py` — 统一到 offloader 框架（~100行）
8. 修改 platform 层 — 注册 NPU offloader（~50行）
9. 配置验证和 CLI 参数集成（~100行）
10. 单元测试（~500行）
11. E2E 测试（~300行）

**预估代码量：~2,050行**
**竞争：1队（明天你好）**
**风险：低** — 框架清晰，不依赖外部 kernel
**性能展示：** "70B 模型原来显存不够跑不了，开了 offload 后能跑" — 效果立竿见鸣

---

#### 候选2：--speculative-config speculative_token_tree（编号19）

**现状：**
- vLLM 上游有完整的 `TreeAttentionBackend`（`vllm/v1/attention/backends/tree_attn.py`，445行）
- GPU 端通过 `qq_bias` 参数传递 tree mask（tree_len x tree_len 矩阵，-inf/0）
- Ascend 的 `eagle_proposer.py`（1712行）有 TODO：`# TODO(wenlong): get more than one token for tree attention`
- Ascend 的 `npu_fused_infer_attention_score` 使用 `atten_mask` 参数（概念上和 `qq_bias` 兼容）
- 完全没有 tree attention 实现

**需要实现：**
1. 新建 `vllm_ascend/attention/backends/tree_attn.py` — AscendTreeAttentionBackend（~600行）
2. 修改 `eagle_proposer.py` — 添加 propose_tree 方法，tree 层级循环（~500行）
3. 修改 `attention_v1.py` — 注册新 backend（~100行）
4. PCP/DCP 兼容（~400行）
5. ACL Graph 兼容（~300行）
6. 动态 shape 处理（~200行）
7. 边缘 case + 错误处理（~200行）
8. 单元测试（~500行）
9. E2E 测试（~300行）

**预估代码量：~3,100行**
**竞争：1队（2-16勇闯Vllm-Ascend）**
**风险：中高** — 需要验证 NPU kernel 是否支持自定义 tree mask（见验证脚本）
**性能展示：** "tree speculative decoding 在 NPU 上首次实现，推理速度提升 X 倍"

---

## 验证计划

在正式认领前，需要在 NPU 机器上运行 `test_tree_attn_npu.py` 验证：

| 测试 | 验证内容 | 通过标准 | 影响决策 |
|------|---------|---------|---------|
| 测试1 | 基本 tree mask 能否被 kernel 接受 | 不报错 | 不通过则无法继续 |
| 测试2 | tree mask 是否正确生效 | 被 mask 的位置输出接近 0 | 不通过需调整策略 |
| 测试3 | 动态 tree mask（batch 推理） | 多请求不同 tree 结构能跑 | 不通过需降级方案 |
| 测试4 | tree mask + paged attention | 和 block_table 兼容 | 不通过需额外适配 |

### 验证结果决策树

```
测试1 通过？
├─ 是 → 测试2 通过？
│   ├─ 是 → 测试3 通过？
│   │   ├─ 是 → 选 token_tree + cpu-offload-params
│   │   └─ 否 → 降级：只支持单 batch tree attention
│   └─ 否 → mask 行为异常，需研究替代方案
└─ 否 → kernel 不支持自定义 atten_mask
    └─ 备选方案：选 cpu-offload-params + gptq
```

## 最终推荐组合

**首选：cpu-offload-params（22） + speculative_token_tree（19）**
- 总代码量：~5,150行
- 总竞争：2队（分散在不同参数上）
- 创新性：高（NPU UVA 调度 + 首次 tree attention）

**备选：cpu-offload-params（22） + quantization gptq（2）**
- 如果 token_tree 验证失败则切换
- gptq 有 GPU 参考实现，确定性高
